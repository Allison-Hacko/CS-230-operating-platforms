The Gaming Room project was centered around a client who wanted to expand their existing game, Draw It or Lose It, which was originally built for Android. The goal was to design a more scalable and flexible version of the application that could potentially run across multiple platforms while maintaining smooth performance and consistency. The software needed to support multiple users, handle game sessions efficiently, and be designed in a way that could adapt to different operating environments, including cloud-based systems.

One thing I think I did well in this project was organizing the design document in a clear and structured way. Breaking down the system requirements, platform comparisons, and architecture considerations made it easier to understand how all the parts fit together. It also helped me think more like a developer planning a real system instead of just writing code without context.

Working through the design document was actually helpful when thinking about the code because it forced me to plan ahead. Instead of jumping straight into implementation, I had to consider things like system architecture, memory usage, and platform differences first. That made the coding process feel more intentional and less random, since I already had a blueprint to follow.

If I could revise one part of my work, I would probably improve the platform evaluation section. I think I could go deeper into comparing cloud and traditional systems with more specific examples of tradeoffs. That would make my recommendations stronger and more realistic for a real client scenario.

I interpreted the user’s needs by focusing on scalability, cross-platform compatibility, and long-term maintainability. Since the client wanted to expand the game beyond a single platform, I made sure to emphasize architectures that would support growth and flexibility. Considering user needs is important because software is only successful if it actually works for the people using it, not just if it is technically correct.

My approach to designing software was to start with the big picture first, then gradually break it down into smaller components like system architecture, storage, and platform requirements. In the future, I would continue using this structured approach, especially focusing more on cloud-native design and modular systems, since they make applications easier to scale and maintain over time.
